# ドキュメント管理方針とテスト基本方針（汎用）

## Docs管理方針（エージェント用含む）

### エージェント運用ルール（必須）

- すべて日本語で対応すること
- 作業前にプロジェクトコンテキストとドキュメント規約を必ず読み込む
- ドキュメントを作成/変更したらドキュメント一覧（INDEX）を更新する
- ドキュメントのファイル名は英語、本文は日本語
- 自動コミット禁止（明示的に指示された場合のみ実行）
- **Issue完了時に必ず `docs/decisions/` へADR（Architecture Decision Record）を作成する**（後から人間が実装経緯を追えるようにするため）

### ADR（決定記録）運用ルール

- Issue単位またはアーキテクチャ上の重要な判断ごとに作成する
- ファイル名: `YYYY-MM-DD-<英語kebab-case>.md`（例: `2026-03-12-fastapi-async-architecture.md`）
- 必須セクション:
  - **背景**: なぜこの判断が必要だったか
  - **決定内容**: 何を採用し、どう実装したか
  - **代替案**: 検討した他の選択肢とその却下理由
  - **結果**: この決定による影響・トレードオフ
- 作成後は `docs/INDEX.md` の `decisions/` セクションを更新する

### ドキュメント命名規則

- ディレクトリ名: 英語の kebab-case
- ファイル名: 英語の kebab-case
- ファイル内容: 日本語

### INDEX更新ルール

- docs配下で新規作成・改名したら必ず更新
- フォーマット:

| File Path | Title / Description (Japanese) | Tags |
| :--- | :--- | :--- |

### Frontmatter（Markdown先頭に必須）

---
title: 日本語のドキュメントタイトル
description: このドキュメントの概要
tags: [タグ1, タグ2]
---

## テスト基本方針

### テストの責務

- Unit: ロジック検証、カバレッジ主担当
- Integration: 外部サービス連携・インフラリソース検証
- E2E: デプロイ後の動作確認

### 必須ルール

- AAAパターン厳守（Arrange/Act/Assert）
- テストケースの説明は日本語で記載
- モック/スタブを使用して依存を分離
- 時間依存のテストは固定時刻で実行
- IaCテストではドライラン（plan）を活用し、実リソース作成を最小限に

### IaCテスト方針

- **静的解析**: linter（pylint, flake8等）、型チェック（mypy）
- **ユニットテスト**: pytest でロジック検証
- **インフラテスト**: `terraform plan` / `cdk diff` 等のドライラン確認
- **統合テスト**: 一時環境へのデプロイ → 検証 → 破棄
